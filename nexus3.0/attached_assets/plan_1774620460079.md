# Nexus Agent — Implementation Plan & Feature Audit

This document records what is **implemented today**, how it maps to the original product intent (`docs/NEXUS_AGENT_PRD.md`), and a **prioritized roadmap**. It was produced by a full-repo review (backend, extension UI, shared libs).

---

## 1. System overview (as built)

| Layer | Role |
|--------|------|
| **Chrome extension (MV3)** | `artifacts/chrome-extension/` — side panel loads built React app from `ui-dist/`. Background service worker, content script for page context and DOM actions. |
| **React UI** | `artifacts/nexus-agent/` — Vite app; `npm run build:extension` emits into `artifacts/chrome-extension/ui-dist/`. |
| **Python API** | `backend/main.py` (FastAPI) — agent sessions, run/stop/observe, intent classification, chat mode, uploads, user input. |
| **Agent core** | `backend/agent/` — `loop.py`, `planner.py`, `executor.py`, `memory.py`, intent + LLM routing. |
| **Services** | Groq, Gemini, OpenAI, OpenRouter clients; Scrapling extraction; `BrowserClient` on the server stubs success for clicks/types; **real** tab execution + `/observe` are implemented in **`sidepanel.js`**, not in the built React bundle (`ui-dist`) unless you add a bridge — see §4. |

**Authoritative runtime path:** User runs `python main.py` (or uvicorn) on the backend; extension is pointed at `http://127.0.0.1:8000` (or configured base URL). Full fidelity (live DOM, screenshots, real clicks) requires the extension to POST `/observe` and execute queued actions (`useAgentBrowserRunner` in React or legacy `sidepanel.js` loop).

### 1.1 First navigation (AI-led; no task-keyword hacks in code)

- **`resolve_search_url` / extension `resolveSearchNavigation`:** Map `search` → native search for the **current tab’s host**, else **Google**. There are **no** `task_context` / “if task contains linkedin” branches — that was removed so **which site opens first** is decided only by the **planner LLM** (`open` with an `https` URL vs `search`).
- **Prompting:** `planner.py` includes **OPEN vs SEARCH**, **FEW-SHOT** first actions, and **`TASK_ANALYSIS`** in the user context so the model aligns with the user’s exact words (e.g. “in LinkedIn” → open LinkedIn Jobs, not Google).

## 2. Backend API surface (verified)

| Endpoint | Purpose |
|----------|---------|
| `GET /api/health`, `GET /api/healthz` | Liveness |
| `GET /api/agent/models` | Lists Groq, Gemini, OpenAI, OpenRouter |
| `POST /api/agent/run` | Start or resume task; resolves model+key with fallback; supports `mode`: `auto` \| `agent` \| `chat` |
| `GET /api/agent/sessions` | List sessions |
| `GET /api/agent/sessions/{id}` | Session detail (logs, actions, extracted data, status) |
| `POST /api/agent/sessions/{id}/stop` | Stop run |
| `POST /api/agent/sessions/{id}/observe` | Extension pushes URL/HTML/text/screenshot + `lastBrowserResult` |
| `POST /api/agent/sessions/{id}/resume` | Resume text/file (e.g. resume upload) |
| `POST /api/agent/analyze-upload` | Analyze files (PDF/images, etc.) |
| `POST /api/agent/sessions/{id}/role` | Job role text |
| `POST /api/agent/sessions/{id}/input` | Structured user input during run |

**Key resolution (`_resolve_model_and_key`):** If the requested provider has no key, fallback order is **OpenRouter → OpenAI → Gemini → Groq** (merged with env vars). This must stay aligned with client-side “can run” checks (see `remember.md`).

## 4. Deep verification: Scrapling, screen capture, logs

### 4.1 Scrapling (HTML → structured page dict)

- **Implementation:** `backend/services/scrapling_service.py` uses `scrapling.parser.Selector` (CSS) to build `extract_page_structure(html)` (forms, buttons, inputs, links, `page_type`, content slice) and `extract_entities(html)`.
- **Where it runs:** `executor.py` after HTTP `get_html`; `loop.py` calls `extract_page_structure(observed_html)` when the session has `current_state.observed_html` from the extension.
- **Verdict:** **Correct** for server-side parsing whenever HTML is present. SPAs that return empty shells over HTTP still look “sparse”; the loop can treat that as sparse and (if screenshot exists) use vision — see below.

### 4.2 Screen capture (tab JPEG → backend → optional Gemini vision)

- **Extension (legacy orchestrator):** `artifacts/chrome-extension/sidepanel.js` → `observeAndSendToBackend` uses `chrome.tabs.captureVisibleTab`, strips the `data:` prefix, and POSTs `screenshotBase64` with HTML/text to `POST /api/agent/sessions/{id}/observe`. **This file is not loaded** when the side panel is the built React app (`manifest.json` → `ui-dist/index.html` only loads Vite bundles).
- **Backend:** `main.py` `observe_page` stores capped JPEG in `current_state["observed_screenshot_jpeg"]`; `memory.py` `to_response()` strips it from JSON so responses stay small.
- **Agent loop:** `loop.py` reads `observed_screenshot_jpeg`, sets `use_vision` when the shot exists and (recent extension failures **or** sparse Scrapling page dict), passes `vision_screenshot_jpeg_b64` into `plan_next_action` → Gemini vision path when configured.
- **Verdict:** **Pipeline is correct end-to-end in code.** The React side panel includes **`useAgentBrowserRunner`** (poll session, execute actions, POST `/observe`) — rebuild extension after UI changes. Legacy `sidepanel.js` does the same when used.

### 4.3 Logs (execution log in UI)

- **Backend:** `store.add_log(...)` throughout `loop.py`, intent, run paths; logs are part of `AgentMemory` and returned in `GET /api/agent/sessions/{id}` as `logs` (timestamp, level, message).
- **UI:** `LogsPanel.tsx` merges `session.logs` and `session.actions`, sorts by time, auto-scrolls; polling via `useGetAgentSession` refetch interval when status is `running` or `waiting_for_input`.
- **Verdict:** **Correct** for the React app: users see server-emitted steps and actions as long as the session is polled.

## 5. Risks and consistency items

1. **Documentation drift:** Update `docs/NEXUS_AGENT_PRD.md` (and `attached_assets` spec) to OpenRouter/OpenAI and extension-centric browser control.
2. **TypeScript API server** (`artifacts/api-server/`): Legacy Node agent loop may not match Python behavior; treat **Python** as source of truth for production.
3. **`pnpm` vs `npm`:** Backend warning references `pnpm`; extension package may use `npm` — follow `artifacts/nexus-agent/package.json` scripts.
4. **Session model vs resolved model:** If run falls back from Groq to OpenRouter server-side, UI “default model” may still show Groq until you add explicit “resolved model” in API responses (future).

## 6. Roadmap (prioritized)

### Phase A — Documentation & DX (short)

- [ ] Align PRD and `artifacts/chrome-extension/README.md` with OpenRouter, no Perplexity, extension execution model.
- [ ] Document env vars: `GROQ_API_KEY`, `OPENROUTER_API_KEY`, `GEMINI_API_KEY`, `OPENAI_API_KEY`, `GROQ_MODEL`, etc.
- [ ] Single “Getting started”: load extension, set API keys, start backend, allow microphone for voice.

### Phase B — UI parity with backend (medium)

- [ ] **Observe pipeline in React (or shared worker):** while a session is `running`, poll `GET /api/agent/sessions/{id}` for new actions *or* mirror `sidepanel.js`: POST `/observe` with HTML from `NEXUS_GET_PAGE_CONTEXT` + `captureVisibleTab`, and execute pending actions in the active tab (required for real browser automation + Scrapling on live DOM + vision).
- [ ] Optional **Run mode** toggle: `auto` | `Always browser` (`agent`) | `Chat only` (`chat`).
- [ ] **User profile** form (name, email, phone, address, …) → `userProfile` on `POST /api/agent/run`.
- [ ] Show **resolved provider/model** in session header when fallback occurs (requires small API + store change).

### Phase C — Product hardening (longer)

- [ ] E2E smoke: health, run with mock, one real navigate task.
- [ ] Rate-limit and error surfacing consistency across providers in UI toasts.
- [ ] Optional: revive remote automation (TinyFish/Playwright) as an **opt-in** path if product needs non-extension environments.

## 7. Success criteria (from PRD, adapted)

- Demo tasks complete or stop safely within step limits.
- No payment / submission without explicit user confirmation (policy in planner/loop).
- API keys never stored server-side permanently; keys in request + optional env for dev.
